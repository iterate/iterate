itx, from a bare RPC socket

itx is the object-capability layer the platform runs on. This is a derivation, not a reference. We'll follow one concrete story — a Node program on your laptop that can run a bit of Swift — and add one thing at a time, each step forced by a concrete failure of the step before, until we arrive at the real design.

The heart of it is two ideas, in order: a capability is a name pointing at a target — either a live stub someone handed you, or a serializable ref to where the real thing lives; and the capability table is the fold of the context's own event stream, which is the only reason any of it survives the Durable Object being evicted. Everything else — calling ergonomics, reaching the target, delegation — hangs off those two.

Step 0

A method call over a socket

Cap'n Web (capnweb) is a bidirectional, promise-pipelined RPC pipe. A server serves one RpcTarget; a client connects, gets a typed stub, and calls its methods as if they were local. Here is the whole thing, and it runs: 

// server.ts — a complete Cloudflare Worker.
import { RpcTarget, newWorkersRpcResponse } from "capnweb";

class Server extends RpcTarget {
  whoami() { return "the itx server"; }
}

export default {
  fetch: (request) => newWorkersRpcResponse(request, new Server()),
};
// client.ts — anywhere.
import { newWebSocketRpcSession } from "capnweb";

using itx = newWebSocketRpcSession<Server>(new WebSocket("wss://…/api/itx"));

await itx.whoami(); // "the itx server" — client calls server, the usual direction

Nothing surprising yet: a typed stub, a method call, using to dispose the session when the block ends. The interesting direction is the other one.

but this is only half of what capnweb can do… the client is always the caller. What we actually want is the reverse: a program on your laptop that the server can reach into.
Step 1

The server calls the client

the cool thing about capnweb Stubs pass as arguments, in either direction. So the client can hand the server a live object, and the server calls methods on it — back across the same socket. Our client is a Node daemon on a laptop that can run a one-off bit of Swift. 
// daemon.ts — a Node program on your laptop. runSwift ACTUALLY runs here:
// `swift -` reads a program from stdin and runs it. that's the whole trick.
import { spawn } from "node:child_process";

const runSwift = (code) =>
  new Promise((resolve, reject) => {
    const swift = spawn("swift", ["-"]);   // `swift -` → read program from stdin
    let out = "";
    swift.stdout.on("data", (d) => (out += d));
    swift.on("error", reject);
    swift.on("close", () => resolve(out));
    swift.stdin.end(code);
  });

const laptop = { runSwift };               // the capability we hand over

using itx = newWebSocketRpcSession<Server>(socket);
await itx.register(laptop);                 // hand the server our laptop object
// server.ts — the server now calls BACK into the laptop:
class Server extends RpcTarget {
  async register(laptop) {
    const out = await laptop.runSwift(`print(1 + 1)`); // executes on the laptop!
    return `your laptop says: ${out}`;             // "your laptop says: 2\n"
  }
}

That's the superpower: laptop.runSwift(...) ran on the laptop, called by code in a Worker. A capability is just an object reference that happens to point across a socket. Everything below is about making that durable, shareable, and nameable.

but this breaks when… the laptop can only offer the one object it passed into that one call, and only the server (its direct peer) can reach it. A real daemon offers several tools (runSwift, runPython, screenshot), wants to add them over time, and — soon — wants other clients to use them. None of that fits "pass one object into one method."
Step 2

provide & invoke — capabilities go dynamic

the one new idea Give the server a registry. provideCapability(name, target) stores a capability by name at runtime; invoke(name, args) calls it. The daemon announces what this laptop can do, by name, and adds more whenever it likes. 
class Itx extends RpcTarget {
  #caps = new Map();

  provideCapability(name, target) {
    this.#caps.set(name, target);
  }

  invoke(name, args) {
    const target = this.#caps.get(name);
    if (!target) throw new Error(`no capability "${name}"`);
    return target(...args);
  }
}

// the daemon announces its tools, by name (runSwift from Step 1):
itx.provideCapability("runSwift", runSwift);
itx.provideCapability("screenshot", () => grabScreen());

await itx.invoke("runSwift", [`print(1 + 1)`]); // "2\n"

Notice what dynamism cost us: in Step 1 we wrote laptop.runSwift(code) — a real method call. Now it's itx.invoke("runSwift", [code]). Dynamic, but clunky. Park that; Step 5 gets the method call back.

but this breaks when… a second client connects. new Itx() runs inside fetch, so every connection gets its own brand-new Map. The laptop provides into its registry; nobody else has a handle to it.
Step 3

A second client

the scenario that breaks it Now a web dashboard opens in your browser. It wants a button that runs Swift on your laptop — so it connects to the server and calls invoke("runSwift", [code]). 
// dashboard.ts — a different client, a different socket.
using itx = newWebSocketRpcSession<Itx>(new WebSocket("wss://…/api/itx"));

await itx.invoke("runSwift", [`print("hi from the dashboard")`]);
// ❌ Error: no capability "runSwift"

The laptop and the dashboard are two separate connections, each handled by a fresh new Itx() in the Worker. The dashboard's registry is empty — the laptop provided into a different one. There is no shared place for them to meet.

the fix is one idea away… we need a single registry that both connections reach — one instance, by name, that the platform guarantees is unique.
Step 4

A Durable Object to live in

the one new idea — and the magic A Durable Object is a single, named, in-memory instance the platform guarantees is unique. Put the Itx inside one, address it by a constant name, and every connection reaches the same registry. The Worker stops minting registries — it just hands each WebSocket session the DO's one Itx. 
// server.ts — the Worker terminates the WebSocket; the DO HOLDS the Itx.
import { DurableObject } from "cloudflare:workers";
import { newWorkersRpcResponse } from "capnweb";

export class ItxDO extends DurableObject {
  #itx = new Itx();             // ONE per DO instance, in this isolate's memory
  itx() { return this.#itx; }    // a METHOD that returns the RPC target (see the note)
}

export default {
  fetch(request, env) {
    const node = env.ITX.getByName("itx");            // constant name → ONE shared instance
    return newWorkersRpcResponse(request, node.itx()); // serve the DO's Itx; the WS lives in the Worker
  },
};

Now the laptop and the dashboard meet in the same place, and the dashboard can run Swift on your laptop: 

// laptop daemon — provides into the ONE shared registry:
itx.provideCapability("runSwift", runSwift);

// dashboard — invokes it; the call lands back on the laptop:
await itx.invoke("runSwift", [`print("hi from the dashboard")`]); // runs on the laptop ✅

That's the rendezvous, and it's the whole point. The dashboard called a function that lives in your laptop's process, routed through the single Itx the DO holds in memory — the laptop's live stub sits in that DO isolate, and the constant name "itx" is what makes "the same place" real.

side note · why itx() is a method, and the WS stays in the Worker We deliberately do not accept and terminate the Cap'n Web session inside the DO. The WebSocket lives in the stateless Worker; the DO only exposes its target through a plain RPC method (node.itx()). Terminating capnweb inside the DO would pin that DO in memory for the life of every open connection. Kenton Varda has signalled capnweb may become hibernatable in future — by making Workers RPC targets hibernatable — so keeping the session in the Worker and the durable state behind an RPC method positions us to adopt that without a rewrite. (It also has to be a method, not a property: workerd can't pipeline calls through property accesses, so node.itx().invoke(…) stays one round trip.)
but two things still nag… first, calling is back to clunky invoke("runSwift", [code]) — we never recovered the method call. Second, the DO is evicted when idle, and on wake new Itx() runs empty. Let's fix the ergonomics first — it's quick — then face durability.
Step 5

Getting the method call back — and who's really proxying

the one new idea Going dynamic looked like it cost us itx.runSwift(code). It didn't — because the client is a naked Cap'n Web stub, and Cap'n Web already turns property access plus a call into a single pipelined message. We never write a client proxy. What we write is the server-side target that receives the name and turns it into an invoke. 
// client — a plain Cap'n Web stub, no wrapper. It sends ["runSwift"] + args itself:
await itx.runSwift(`print(1 + 1)`);  // → one pipelined call, path ["runSwift"] → "2\n"
// server — the registry is DYNAMIC, so the served target can't be a fixed class.
// serve a proxy whose unknown names become invoke():
function handle(core) {
  return new Proxy(core, {
    get(core, key) {
      if (key in core) return Reflect.get(core, key);  // provideCapability, invoke, describe…
      return (...args) => core.invoke(key, args);       // unknown → invoke("runSwift", [code])
    },
  });
}

// the DO serves the wrapped core:
itx() { return handle(this.#itx); }

itx.runSwift(code) feels native again — but notice the proxy is on the server, not the client, and it's pure sugar over the same dynamic invoke. (This one-level version works because Cap'n Web walks an RpcTarget by reading its members directly. The deep version below trips a subtler rule.)

but this breaks when… the capability is a whole SDK and you want itx.slack.chat.postMessage({ channel, text }) — a deep path, ideally typed against the official Slack package. Cap'n Web pipelines the whole path from the client just fine; the breakage is on the server, where traversing past the first segment needs more than a get trap.
Step 6

Deep paths & the Slack SDK

the motivation Mount the official @slack/web-api client as one capability and call straight into it — itx.slack.chat.postMessage(…) — with the SDK's own types. The client writes nothing new: Cap'n Web accumulates ["slack","chat","postMessage"] locally (zero round trips) and sends one pipelined call. The server mounts the cap at a path, resolves the longest registered prefix, and replays the remainder onto the target. 
// provide the real Slack SDK as ONE capability, mounted at "slack":
import { WebClient } from "@slack/web-api";
itx.provideCapability("slack", new WebClient(token));

// client — a naked stub typed as { slack: WebClient }, so the editor autocompletes the
// real Slack API. Cap'n Web sends ["slack","chat","postMessage"] + [msg] in ONE message:
await itx.slack.chat.postMessage({ channel: "C123", text: "hi" });

// server — resolve the LONGEST registered prefix, replay the rest onto the target:
invoke(path, args) {
  const { entry, remainder } = resolveLongestPrefix(this.#caps, path); // "slack" wins
  return replayPath(entry, remainder, args); // webClient.chat.postMessage(msg)
}

itx doesn't know a thing about Slack; it routes a path. And there is no client-side path proxy — Cap'n Web's stub already is one. (Production matches this exactly: the browser, Node, and the REPL all hold a plain newWebSocketRpcSession<ItxHandle> stub; the path proxy in path-proxy.ts runs server-side, inside the handle.)

the gotcha that cost a day The dynamic server target has to answer for names it has never seen. Three non-obvious rules make that work (all verified in min-dynamic-target.mjs in this folder):
  • getOwnPropertyDescriptor is load-bearing, not just get. Server-side Cap'n Web does Object.hasOwn(value, segment) before reading each segment; without a descriptor trap every segment reads as absent and the chain dies at ".chat of undefined" — which is exactly why our first draft wrongly concluded deep server-side dispatch "doesn't work."
  • The target must be function-typed — a Proxy over a plain function, not over an RpcTarget. Cap'n Web classifies an rpc-target by its prototype and rejects fabricated "instance properties" (it even flags the real verbs).
  • Retain (dup) provided live stubs — Cap'n Web disposes argument stubs when the provide call returns.
what falls out for free Because the longest registered prefix wins, you can shadow a single method deep inside another capability. Provide at ["slack","chat","postMessage"] and that one call resolves to your override; slack.users.list and everything else still resolves to the original slack client. One path, one rule, and surgical overrides drop out: 
// wrap just chat.postMessage with rate-limiting; leave the rest of Slack intact.
itx.provideCapability("slack.chat.postMessage", rateLimited(slack.chat.postMessage));
await itx.slack.chat.postMessage(msg); // → your wrapper (longest prefix)
await itx.slack.users.list();          // → the original WebClient (prefix "slack")
but this breaks when… the DO is evicted (we kept deferring it). And we still haven't said what a capability is — the laptop's runSwift is a live function in a process that may have closed its lid; the Slack client is a live object too. Some capabilities can't survive an eviction at all. We have to name the kinds.
Step 7

Two kinds of target: live stub and sturdy ref

the one new idea — and half the whole design A capability is a name pointing at a target, and a target is one of exactly two things:
  • A live stub — an in-memory object someone connected and handed in. The laptop's runSwift is one: it grants its power by being held, and lives only as long as that laptop's connection.
  • A sturdy ref — plain, serializable data that says where the implementation lives. Our example: a capability defined in the project's own worker. The ref is just a description; the platform builds and runs it on demand. 
// "formatInvoice is defined in the project's worker." encode WHERE, as data:
const formatInvoice = {
  type: "rpc",
  worker: { type: "source", source: { repo: "acme/app", commit: "9a1c…", path: "tools.ts" } },
};
itx.provideCapability("formatInvoice", formatInvoice); // pure data — survives anything

// the discriminator is structural AND order-sensitive (isCapabilityAddress):
function isRef(cap) {
  if (!isPlainObject(cap)) return false;   // MUST come first — see below
  return cap.type === "rpc";            // the only serializable kind
}

class Itx extends RpcTarget {
  #caps = new Map();        // name → { kind, ref }  (the durable record)
  #liveStubs = new Map();   // name → stub          (in-memory, connection-bound)

  provideCapability(name, capability) {
    if (isRef(capability)) {
      this.#caps.set(name, { kind: "rpc", ref: capability });   // always available
    } else {
      this.#caps.set(name, { kind: "live", ref: null });        // no serializable form
      this.#liveStubs.set(name, capability);                  // the stub lives here
    }
  }
}

Why check plainness before reading .type? A capnweb stub proxies every property access — reading .type off the laptop's live stub returns a truthy pipelined stub and would misclassify it as a ref.

The asymmetry is the point. runSwift needs your laptop online; formatInvoice is code in a repo, so it's always available and is entirely plain data. That difference is exactly what survives the next step — and what can't.

but this breaks when… the DO is evicted (still). We now know why the in-memory table can't be persisted wholesale — half of it is live stubs with no serializable form. But the records can be saved. We need somewhere durable to put them.
Step 8

Durability: the table is the fold of the context's stream

the keystone The Durable Object already owns a persistent event stream. Make that the source of truth, not the Map. provideCapability becomes a pure append of one capability-provided event; the capability table is the fold of the stream. On wake, the DO re-reads its stream and re-derives the table — the Map is now just a cache. 
// THE FOLD — pure, no I/O, the heart of the system (reduceItxEvent):
function reduce(state, event) {
  switch (event.type) {
    case "capability-provided": {
      const name = event.payload.path.join(".");
      const { kind, address } = event.payload;       // kind: "live" | "rpc"
      return { ...state, capabilities: { ...state.capabilities, [name]: { name, kind, address } } };
    }
    case "capability-revoked": {
      const caps = { ...state.capabilities };
      delete caps[event.payload.path.join(".")];
      return { ...state, capabilities: caps };
    }
    default: return state; // unknown/old shape → unchanged. replay must never wedge.
  }
}

class Itx extends RpcTarget {
  #stream; #state = { capabilities: {} }; #offset = 0;

  async #catchUp() {
    const events = await this.#stream.read({ afterOffset: this.#offset });
    for (const e of events) { this.#state = reduce(this.#state, e); this.#offset = e.offset; }
  }

  async provideCapability(name, capability) {
    const live = !isRef(capability);
    if (live) this.#liveStubs.set(name, capability);  // stub: still in-memory only
    await this.#stream.append({
      type: "capability-provided",
      payload: { path: [name], kind: live ? "live" : "rpc",
                 address: live ? null : capability },  // the ref is the durable part
    });
    await this.#catchUp();  // read-your-writes: observe your own event through the one door
  }
}

Now the two kinds from Step 7 pay off, asymmetrically — the honest, important part:

  • formatInvoice is just data, so it goes straight into the event. After an eviction the fold rebuilds it perfectly; still fully callable.
  • runSwift can't be serialized. We still append the event (with address: null) so the record survives, but the stub died with the laptop's connection. It comes back offline: the entry exists, #liveStubs has nothing, and calling it throws CapabilityOfflineError ("the laptop must reconnect and re-provide").

Two invariants drop out. The fold is defensive — an unknown or malformed payload returns the state unchanged (default: return state), so replay can never wedge on a pre-deploy event shape. Read through the one door — after appending you don't mutate #state; you #catchUp through the same fold replay uses, so write-time and replay-time can't drift. The record and the state cannot disagree, because events are the only writes.

but this breaks when… invoke has to actually run formatInvoice. A ref is just { type:"rpc", worker:{…} } — turning it into something callable means building the project's worker and getting an entrypoint. That's effectful, environment-specific, and doesn't belong in the pure core.
Step 9

dial: restoring a ref — run the project worker

the one new idea The core never touches env or the worker loader. It depends on one injected function, dial, that restores a sturdy ref to something speaking call({ path, args }). Live caps borrow their stub; ref caps get dialed. Both converge on one PathCallable. 
// dial lives OUTSIDE the core. for a "defined in the project worker" ref it
// builds that worker from its source and grabs an entrypoint (Worker Loader):
function makeDial(env, loader) {
  return (ref, attribution) => {
    const { worker } = ref;

    if (worker.type === "binding")                       // a plain env binding (e.g. AI)
      return wrap(env[worker.binding]);

    if (worker.type === "source") {                      // built from the project repo
      const isolate = loader.get(cacheKey(worker.source), async () => {
        const { mainModule, modules } = await build(worker.source); // compile repo@commit/path
        return { mainModule, modules, compatibilityDate: "2026-04-27" };
      });
      const entry = isolate.then((w) => w.getEntrypoint());
      return { call: async (input) => replayPath(await entry, input.path, input.args) };
    }
    // …loopback, durable-object (see addendum)
  };
}

// inside the core, invoke is now two cases and one uniform call:
#borrow(entry) {
  if (entry.kind === "live") {
    const stub = this.#liveStubs.get(entry.name);
    if (!stub) throw new CapabilityOfflineError(entry.name); // the laptop is offline
    return stub;
  }
  return this.deps.dial(entry.address, { capabilityPath: entry.name }); // build + run the worker
}

So itx.formatInvoice(order) resolves the ref, the platform builds the project's worker (cached by content, per origin), grabs its entrypoint, and calls it — your repo's code, run on demand, with no live connection required. The core owns structure (the fold, the discriminator, dispatch); dial owns reach (what can be built and called, gated at call time). The core stays testable with a fake dial and an in-memory stream — no workerd.

but this breaks when… you want a capability that isn't on this context. A throwaway agent run should see the project's slack and formatInvoice without re-providing them — and should be able to override one of them for itself without touching the project. We need contexts that build on other contexts.
Step 10

extend & super — the context chain

the one new idea A context can have a parent. extend() mints a cheap child — it's Object.create(parent) for capability tables. The child's own provides shadow the parent's; on an invoke miss it delegates the whole path up the chain; and super reaches the parent's version of something the child has shadowed — exactly like JavaScript.

The motivation is the throwaway scope. An agent run, an MCP session, a request: each wants its own context that sees everything the project offers, can add run-specific capabilities, and can override one or two — all without mutating the project context that everyone shares. 

// a per-run child context. it inherits the project's caps for free.
using run = await itx.extend({ name: "agent-run-42" });

await run.formatInvoice(order);     // inherited from the project — a miss here, found up the chain

// shadow just chat.postMessage for THIS run — to log every message it sends:
run.provideCapability("slack.chat.postMessage", async (msg) => {
  log("run-42 posting:", msg);
  return await run.super.slack.chat.postMessage(msg); // the PARENT's real Slack
});

await run.slack.users.list(); // still the project's Slack (deep shadow is surgical)
// dispatch: longest local prefix, else delegate the whole path to the parent.
async invoke(path, args, origin) {
  await this.#catchUp();
  origin = origin ?? { ref: this.deps.contextRef, address: this.deps.selfAddress };
  const r = resolveLongestProvidedPrefix(this.#state.capabilities, path);
  if (r) return await this.#borrow(r.entry, origin).call({ path: r.remainder, args });

  const parent = this.deps.parentItx();
  if (parent) return await parent.invoke({ path, args, origin }); // delegate, carry origin
  throw new Error(`no capability "${path[0]}"`);
}

Three things compose here. Inheritance is just the miss-delegates-up rule. Shadowing is just "your entry resolves before the parent's" — and because dispatch is longest-prefix, a child can shadow one deep method (the slack.chat.postMessage above) while everything else under slack still falls through to the parent. super is the same delegation done explicitly, so a shadow can wrap the thing it replaced. The chain bottoms out at a code-rooted "defaults" context where the platform's built-ins (fetch, streams, secrets, …) live.

A context's identity is its stream coordinate: the ref <namespace>:/<path> — prj_x:/ for a project, prj_x:/agents/run-42 for that child. One string is the identity, the stream coordinate, and the host DO's name (the constant "itx" from Step 4, grown up). Self-describing: hold the ref and you can find and dial the context, no directory. Parentage is the first event in the stream — a context-created { name, parent } birth certificate, folded like everything else.

but this breaks when… you notice you've hand-rolled "fold a stream with a checkpoint, catch up incrementally, serialize batches" — #catchUp, #offset, read-your-writes — and now per-context for a whole chain of them. That machinery is subtle and already exists. Reimplementing it is exactly how the record and the state start to disagree.
Step 11

Itx is a StreamProcessor

the punchline A capability table that is the fold of a stream, with a checkpoint, is a stream processor. So Itx doesn't use one — it is one. reduce becomes the processor's hook. The checkpoint becomes a disposable cache of the fold (lose it, re-derive from the stream). provide/revoke append to the stream and catch up through the processor's own ingest. 
class Itx extends StreamProcessor<typeof ItxContract, ItxDeps> {
  readonly contract = ItxContract; // declares the event vocab + state schema

  // the fold is the processor hook — literally the pure function from Step 8:
  protected override reduce(args) { return reduceItxEvent(args.state, args.event); }

  async provideCapability(input) {
    await this.#materialize();              // lazily consume the stream once
    await this.#append("capability-provided", { /* path, kind, address, owner, meta */ });
  }

  async #append(type, payload) {
    await this.ctx.stream.append({ type, payload });
    await this.#catchUp();  // read from checkpoint, ingest forward — contiguous under concurrent writers
  }
  // invoke / describe / revoke from Steps 9–10, reading this.state (the folded state)
}

This is the real design. The processor base hands you, for free, the things we were about to hand-roll badly: checkpoint storage, batch serialization (a later batch never starts mid-earlier-batch), and replay-from-history without re-running side effects. The contract makes "the context's state is the fold of these events" a typed statement, and the invariant — the record and the state cannot disagree, because events are the only writes — is enforced by construction. The Durable Object from Step 4 grows up into ItxDurableObject: one host for every context, its itx events interleaved on the context's own stream.

And the laptop daemon from Step 1 is exactly how the real platform works: a Node process (or a browser, or an agent run) connects, provides live capabilities, mounts SDKs at paths, and the platform routes calls to them — until it disconnects, when they go offline and the durable refs carry on.

What the small core actually is

The whole thing reads top-to-bottom as one story: the server can call the client; make capabilities dynamic (provide/invoke), shared (a Durable Object), ergonomic (a proxy), and deep (paths → longest-prefix); a capability is a name → a target (live stub or sturdy ref); the table is the fold of the context's stream; dispatch borrows the live stub or dials the ref; a miss climbs to the parent. Concretely it's a few hundred lines:

Piece name ~lines
The fold (single source of truth) reduceItxEvent ~60
live-vs-ref discriminator isCapabilityAddress ~25
The one write path provideCapability / revoke ~80
Borrow-or-dial + the dial type #borrow ~30
Dispatch + chain delegation invoke ~40
Server-side path proxy + longest-prefix + replay PathProxy / resolveLongestProvidedPrefix / replayPathCall ~50
Chain-merged, shadow-aware view describe / extend / super ~45
Write/consume seam (read-your-writes) #append / #catchUp / #materialize ~50

The test for "core vs addendum": does removing it change what reduce computes, or what a live-vs-ref target means? If no, it's an addendum item — and everything that survives that test is the small core that tells one coherent story without a single line of the rest.

The addendum: everything we layered on, and why

All of this is real and lives in the actual files. None of it changes the inner model — the core is complete and correct without every item here. Each entry is the concrete failure it buys you out of.

The server-side proxy's sharp edges — then, reserved segments

The Step-6 server-side proxy (production's PathProxy, reached through the remote handle) needs a then trap reading as undefined (so a path node is never mistaken for a thenable mid-chain), and it must refuse protocol-level segments (__proto__, apply, call, dup, Cap'n Web stub controls) on intermediate path nodes — the same reserved set guards both the proxy and the server-side replayPathCall.

Without it: a crafted path reaches a prototype-pollution / RPC-control name, or await trips on a path node. Pure protocol hygiene over the same path dispatch.

Live-stub retention — dup(), onRpcBroken, the disconnect tombstone

capnweb disposes argument stubs when the call returns, so the laptop's stored runSwift stub must be dup()'d (and plain-object providers deep-walked to dup every member); teardown appends a capability-disconnected event so describe() shows it offline.

Without it: a stored live capability is silently dead after its first call returns. It never touches the fold or dispatch — pure connection-lifetime plumbing.

The handle is more than the proxy

The Step-5/6 proxy is the kernel; the real ItxHandle also carries built-in members (projects, streams, fetch, extend, super) that don't fall through to invoke, plus reserved-name gating, and it's what the Worker serves (a thin wrapper dialing the DO node's itx()).

Without it: you call invoke by hand and lose the built-ins. Authority is the handle itself — auth happens once, at connect.

The ref taxonomy and dial's reach

Beyond source (built from a repo), a ref's worker can be binding (an env binding like AI), loopback (a first-party entrypoint), or durable-object; dial handles Worker-Loader isolate caching (by content, per origin), facets for stateful caps, and allowlists gating what's reachable.

Without it: only one ref shape works. The core only knows "ref → PathCallable via the injected dial."

Dial allowlists, injected attribution, and spoof-proofing

Reach is gated at invoke time, not provide time, so widening deployment config never strands old provides. Every dialed target is handed injected props (capabilityPath, context, projectId) that overwrite caller-supplied props.

Without it: any handle holder could provide a cap that hijacks a loopback into another project, or spoof its origin. Authorization is established once, at connect; this keeps it from being escalated.

A context is its stream coordinate

Identity is the ref <namespace>:/<path> — which is also the host ItxDurableObject's name and its dial address. One host runs every context (the Step-4 DO, generalized); there are no synthetic context ids, no directory table, no reserved stream segment. resolveItx needs no catalog and no special cases because every ref is self-describing by construction.

Without it: you'd need a directory to turn an id into an address (and threading that address around is exactly the class of bug self-describing refs retire). It's where a context lives and how you re-find it — orthogonal to what it does.

Hibernation, deferred

The session terminates in the stateless Worker, never inside the DO, and the DO exposes its target through an itx() method (Step 4). Today this just keeps the DO from being pinned per-connection; it also positions us to adopt capnweb hibernation if/when Workers RPC targets become hibernatable (Kenton Varda has signalled it's on the table).

Without it: long-lived connections pin DOs in memory and foreclose a future where idle itx sessions hibernate. Doesn't touch the capability model at all.

fetch as a shadowable capability — one pipe, two doors

Egress is just a default fetch cap. Bare fetch() inside loaded isolates (like the project worker) and itx.fetch() both route through it; a live shadow on a child context intercepts both; the terminal pipe substitutes getSecret(…) placeholders outside the isolate so code never sees secret material. It's the deep-shadowing trick from Step 6/10 applied to egress.

Without it: no egress interception, no secret hygiene. But it's an application of provide/shadow/dispatch — not a new mechanism.

Origin threading

invoke carries origin: { ref, address } up the chain, so an inherited capability's bare fetch() dials back through the originating context's chain — a child's fetch shadow catches its own isolates' egress, not the parent's. Nodes set it; handles never forward it.

Without it: shadows on child contexts wouldn't intercept calls made through inherited capabilities. Dispatch works without it; context-scoped dial-back doesn't.

Read-your-writes hardening

The Step-8 #catchUp is correct single-threaded; the production version tracks an #appendCount and loops a single-flight sync until one that started at or after the current count completes, surviving concurrent external writers interleaving offsets.

Without it: a provide could fail to observe its own event under concurrency. Conceptually identical to Step 8 — "observe your own write through the one door," made robust.

Two RPC systems — Cap'n Web pipelines paths, Workers RPC doesn't

Cap'n Web's RpcTarget is a Workers RpcTarget in workerd, but the two systems behave differently on property paths. Cap'n Web pipelines a whole dotted path from a naked stub into one call — so itx.slack.chat.postMessage(…) needs no client proxy; the server-side dynamic proxy only has to answer Object.hasOwn for each segment (its getOwnPropertyDescriptor trap, Step 6). Workers RPC (the DO stub, the dialed parent) does not pipeline through property accesses — which is why the DO exposes its core via an itx() method (Step 4's note) and why itx.project.processor.snapshot() is made to work by replayPathCall awaiting each segment server-side. A DO's in-memory live-stub table can't cross RPC; reconnecting replays the stream and marks live entries offline.

Without it: calls hang or fields read as undefined across the Workers-RPC boundary. These shape the wiring, not the model.

Script execution (codemode)

A capability can be a whole script: script-execution-requested (enqueued: true) runs an async (itx) => … program in a loaded isolate; the runner appends script-execution-completed. The synchronous /api/itx/run door writes the identical two-event record, converging both modes on one event vocabulary.

Without it: no agent code execution. But it's a second consumer of the processor machinery, bolted alongside the capability table — the table doesn't depend on it.

Access non-escalation & the platform defaults root

A project-context handle's access is always exactly its own project, regardless of what props claim; a global (admin) handle must narrow through projects.get(). The chain bottoms out at a code-rooted, read-only platform context — its defaults (fetch, streams, secrets, repos, workspace, ai) are loopback refs dialed via dialCodeContext, so they update the instant you deploy.

Without it: a context could reach sibling projects, and platform defaults couldn't be edited in one place. The chain mechanism (Step 10) is unchanged; this is its root and its guard rails.

Metadata projection — instructions, types, owner, updatedAtMs

Convention fields lifted to the top level by describe() (human-readable instructions, TypeScript types), plus provenance bookkeeping. Arbitrary meta is preserved verbatim on the event.

Without it: a less legible describe(). Pure metadata — it never affects resolution.

That's the whole idea. The inner core — a name → a live-or-ref target, a table that's the fold of the context's stream living in one Durable Object, a proxy that makes calling it feel native, paths that mount whole SDKs and shadow deep, borrow-or-dial, climb to a parent on a miss — stands entirely on its own and reads as one story. Everything in the addendum is reach, durability hardening, runtime quirks, and security: real, necessary, and none of it load-bearing for understanding what a context is.
