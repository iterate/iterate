itx — the iterate context, built up from a Map

This page constructs the itx system one concept at a time, starting from ten lines of JavaScript and ending at the real implementation in apps/os/src/itx/. A miniature but faithful core — same fold, same dispatch, same chain — runs inside this page and powers every demo. Where the page simplifies, it says so inline; everything stated as the real API is checked against types.ts (the design of record), itx.ts (the core), and examples.ts (the live recipe catalogue).

Section 0

Why: authority should be held, not configured

The platform is full of actors that need to offer each other abilities. An agent needs the project's repos and an AI model. A script needs to read a stream. Your laptop wants to lend its Swift toolchain to an agent for an afternoon. A browser tab wants to intercept every outbound HTTP request of one session. Code committed to a project's repo wants to be callable as a tool. A third-party server across the internet wants in too. Wiring these pairwise — Slack-to-agent, laptop-to-REPL, repo-to-worker, each with its own registration protocol, its own auth, its own discovery — grows quadratically and every edge invents its own permission model. The classical fix is configuration: central registries, scopes, API keys sprinkled through env vars. itx takes the capability-systems answer instead: authority is held, not configured. There is one kind of node — a context, an addressable object that holds named capabilities — and one kind of reference to it — an itx handle. Whoever holds a handle can call what the context holds; providing, calling, describing, and overriding are the same four verbs for every actor in the picture. Auth happens once, at connect; after that, which context your handle points at is the entire permission model.

Each arrow above is the same mechanism. By the end of this page you will know it completely: a path-keyed capability table (§1–2), replayed from an event journal (§3), holding live stubs or serializable addresses (§4), arranged in a prototype chain whose root is code (§5), dispatched through one effectful function (§6), identified by stream coordinates (§7), with code itself as just another capability kind (§8), self-describing throughout (§9) — and a doctrine for building the next thing the same way (§10).

Section 1

A table of capabilities

Strip everything away and the seed of the whole system is ten lines: a Map from a name to a thing, a verb to put things in, and a verb to call them.

That is already recognizably itx: provide doesn't care what the thing is or where it came from; invoke doesn't care who's asking; the table is the single point of indirection between them. Everything that follows is this table acquiring the properties production needs — a wire-safe calling convention, durability, inheritance, reach, identity, and self-description — without ever stopping being a table.

Two things are wrong with the seed, and fixing them produces the next two sections. First, the value side is a bare function — real capabilities are whole API surfaces (slack.chat.postMessage), so we need a calling convention that carries structure (§2). Second, the table lives in memory — restart and it's gone, and there is no record of who put what there. The fix is not "add a database"; it's that the table stops being written at all and becomes a fold over events (§3).

Section 2

One calling convention: call({ path, args })

The kernel knows exactly one way to call anything. Every dispatch in the system — every capability, every transport, every host — bottoms out in:

Two pure functions convert between dots and data, and they are the only places paths are interpreted:

  • Dots become data — PathProxy (consumer side, path-proxy.ts). A callable JS Proxy where property access accumulates a path locally — zero round trips — and the terminal call fires once with the accumulated PathCall. itx.slack.chat.postMessage(msg) becomes { path: ["slack","chat","postMessage"], args: [msg] }. The proxy reads then as undefined so await never mistakes a mid-chain node for a thenable.
  • Data becomes dots — replayPathCall(target, { path, args }) (receiver side). Walk the segments on a concrete object and call the terminal method on its parent (receivers preserved — Workers RPC methods can depend on this). An empty path calls the target itself as a function. A shared RESERVED_PATH_SEGMENTS set (__proto__, constructor, then, call, dup, describeItx, …) is filtered on both sides; the replay is the authoritative gate, because invoke is public and a path can be hand-built.

Dispatch resolves the longest provided prefix of the call path and hands the remainder to the target. Because the convention is universal, any shape plugs in with zero client wrapper — the choice of how a path maps onto an object lives at the edge where the concrete object is, never in stored data:

Section 3

The journal is the only authority

Now make the table durable — by never writing it. In itx, provideCapability and revokeCapability do not mutate a table; they append events to the context's journal, an ordinary event stream. The table you saw in §1 is the fold (reduce) of that stream. The complete event vocabulary (contract.ts):

What "the journal is the only authority" buys, concretely:

  • State is a fold; the checkpoint is a disposable cache. The processor's storage holds { offset, state } — a memo of reduce over the journal. Delete it and replay rebuilds it, bit for bit. The record and the state cannot disagree, because events are the only writes.
  • Exactly-once is a property of the fold, not of delivery. The reducer takes the first birth certificate and ignores any later one; duplicate deliveries are inert by offset bookkeeping. No idempotency keys as a correctness mechanism, no ON CONFLICT DO NOTHING.
  • No initialize RPC. Creating a context is one append: mint an itx_… id, append context-created { id, name, parent: { id, address } } — the birth certificate, the journal's first event — and return. Nothing touches the new node; it materializes lazily by consuming its own journal on first dispatch.
  • "Audit log" stops existing as a separate concept. Anything that needs to be auditable is an event, because events are the only writes. The journal is an ordinary readable stream — every stream viewer shows it.
  • Live provides journal the EVENT too. The record outlives the session (the stub itself stays an in-memory instance field, §4); after a replay the entry shows up as registered-but-disconnected.

Section 4

Live vs durable: stubs and addresses

What can sit in the table? The CapabilityTarget union has exactly two arms, and the discrimination is structural:

An address is a plain object — prototype Object.prototype or null — carrying type: "rpc" | "url". Everything else is live. The plainness check must run before any .type probe: property access on a capnweb stub returns a truthy pipelined stub, so reading .type first would misclassify every live capability. (A plain object with a string type that isn't "rpc"/"url" is rejected loudly as a malformed address — a typo must not register as an offline live cap.)

Live: a stub you are holding

  • Inbound: something connected to this context and handed it a stub. The capability exists exactly as long as that connection does — live caps die with the session and come back when the provider reconnects and provides again. There is no durable delivery on live stubs: offline means offline.
  • The stub lives in the core's in-memory instance table (dup-retained), never in the journal — but the provide event is journaled, so after replay the entry survives as registered-but-offline. describe() reports live entries with connected: true | false; calling an offline one throws CapabilityOfflineError ("…the provider must reconnect and call provideCapability() again").
  • Session teardown appends capability-disconnected — record only; the fold ignores it (connectedness is computed from the live table, never from events).

Durable: an address — this realm's sturdy refs

The serializable kinds are the realm's SturdyRef format (in Cap'n Proto's sense): pure names that can be stored in a journal and restored to a live object on demand. Crucially they grant nothing by possession — they are names within a trusted realm, not bearer tokens; restoring is itself gated (authority is which context your handle points at, checked at connect). The one deliberate bearer-form exception is the HTTP share URL (§9), at the edge only. Real address JSON, as it sits in journals:

Context nodes themselves are addressed with the same data structure — an address whose worker ref is a Durable Object is "how to dial the node that owns this identity" (§7). One data structure; everything is a use of it.

What provideCapability returns: a provision handle { revoke(), [Symbol.dispose] }. Dispose auto-revokes only live provides — a live cap dies with its session anyway, while a durable provide must outlive the session that created it (session teardown disposes every returned handle, so a revoking disposer would silently undo durable provides). Also: provide-time validation is structural only (path shape, address well-formedness). Whether an address is reachable is the dial's authority and surfaces at first call (§6).

Section 5

The chain: extend, shadow, super

Contexts form a prototype chain. Behaviorally, a context is a prototype-chain object whose properties are capabilities: lookup walks child → parent with shadowing, writes land on the node your handle points at, and extend() is Object.create(parent) — it mints a cheap child context with its own journal (birth certificate first, §3), whose capabilities shadow the parent's and whose misses delegate the whole path up the chain.

The rules — all consequences of the chain, not separate mechanisms:

  • Shadowing is per path. Entries live at paths; provideCapability({ path: ["slack","chat","postMessage"], … }) shadows ONE subtree of an inherited cap while every other slack.* call falls through. Longest provided prefix wins per context; no prefix → delegate the whole path to the parent.
  • Revoke resurfaces. Revoking a shadow makes lookup reach the chain again, so the inherited entry comes back — by construction. Revoking an inherited entry is refused with a hint ("provide your own on this context to take its place; the original stays on the parent"): claiming to revoke something that lives on another node would lie.
  • itx.super is middleware's "call next()". A handle on the parent context: a fetch shadow delegates to the unshadowed pipe via itx.super.fetch(request). (One priced papercut: super is a reserved word, so itx.super works but destructuring it does not.)
  • Delegation carries origin — { id, address } of the context the call started at, set by delegating nodes, never by handles. It is the chain's trusted identity channel: attribution, context-scoped capabilities (the workspace), and origin dial-back (§6).
  • describe() is the merged chain view. Own entries first (no provenance field — they are yours), then ancestors' entries, each stamped from: <context id>. Suppression is exact-match only: a path shadow like sdk.chat.postMessage does not hide the parent's sdk, because that entry is still live for every other path.

The root of every chain is code

The chain does not end in data. Every project context's parentItx is the defaults — PlatformContext (platform-context.ts), a read-only loopback WorkerEntrypoint that answers the same four-verb context protocol as every node: describe and invoke from a literal list in code, provideCapability / revokeCapability refused ("The defaults are read-only — they ship with the deploy"). The defaults every project inherits: ai, fetch, streams, secrets, repos, workspace, worker — each with its own instructions, so even the code-rooted defaults self-describe. In describe() they appear as from: "defaults" (DEFAULTS_DESCRIBE_FROM, types.ts — the internal platform:project chain id never leaves the chain).

Shipping a new default is a deploy, not a migration: the chain sees it immediately; journaled rows shadow it; revoking a shadow resurfaces it. There is no defaults mechanism — only the chain. And because the parent link is dialed in-process (loopback), default dispatch pays no Durable Object hop.

Section 6

The dial: the one effectful function

The core is pure structure: entries, the fold, longest-prefix dispatch, chain delegation. Everything effectful is injected as one function — the dial:

The core owns structure; the dial owns reach. Every invoke, whatever it looks like at the callsite, ends in one call({ path, args }) against something the dial resolved:

Per address kind, the dial (dial.ts) does:

  • binding — gate on the bindings allowlist, take the concrete object off env, wrap it in-process with the §2 replay (a binding doesn't speak the convention; wrap it where it is real).
  • loopback — gate on the loopbacks allowlist, instantiate the named export with { ...address.props, ...injected }. Injection wins by spread order: the dial-injected { capabilityPath, context, projectId } overwrite anything provider-supplied, which is the spoof-proofing — a provider can never point a dialable loopback at someone else's project. First-party loopbacks implement call themselves.
  • source — materialize an isolate via the Worker Loader (§8), wired by wireIsolateEnv with env.ITERATE (an itx scoped to the origin context) and globalOutbound = ProjectEgress (Law 5). exportType: "durable-object" sources become facets of the hosting DO with their own private SQLite.
  • durable-object — gate on the (default-empty) namespace allowlist, then getByName("itx:" + projectId + ":" + name) — instance names are scoped under the owning project, so two projects naming the same instance get disjoint objects.
  • url — cross to the stateless UrlDial loopback entrypoint, which opens one Cap'n Web WebSocket session per call against the remote main and disposes it. Law 7: Cap'n Web terminates in the stateless worker, never in a DO.

Reachability errors surface at the capability's first call, never at provide time (provide is structural only). That asymmetry is deliberate: a deployment that widens its allowlists never strands rows that were provided before the widening.

Origin dial-back, the dial's subtlest consequence: a source capability's isolate is wired to the originating context, not the definition's home. Invoke an inherited source cap through your extension and its bare fetch() climbs your chain — shadow fetch on the extension and the inherited capability's outbound traffic lands in your shadow, while siblings and the parent are untouched. (Context addresses are dialed by a separate kernel restore, dialContext in journal.ts — deliberately not allowlist-gated, because only kernel code writes context addresses.)

Section 7

Identity is a stream coordinate

Where does a context live? Its journal is an ordinary event stream, and streams are keyed by (namespace, path). Journals live at <host base>/itx[/<child-id>] in the owning project's namespace:

  • The DO name IS the coordinate. The generic context host (ItxDurableObject) holds no configuration: its Durable Object name encodes the coordinate verbatim — <namespace>:<journalPath> — so identity, journal ref, and self-address are projections of the name, derived and never configured. Parentage folds from the birth certificate; descriptor() derives from state. Address, journal, and identity are three parses of one string.
  • itx is a reserved stream path segment. Domain code and user streams may not write under it — the user-facing append doors refuse with one clear error (assertStreamPathDoesNotClaimItxSegment). Journals stay readable everywhere: they are ordinary streams, and every stream viewer shows them.
  • The D1 row is a directory, never the authority. Bare itx_… refs (reconnects, /api/itx/run, isolate props) resolve to their coordinate through the itx_contexts D1 catalog — D1 in its sanctioned role (a directory), while parentage and state fold from the journal.
  • The project context is code-born. It is hosted by the Project DO (addressed by the project id, binding PROJECT), its journal is (projectId, "/itx"), and it has no birth certificate — its identity derives from its host's name; its parentItx is the defaults. Only extended children carry context-created as event #1.
  • Chain depth ≠ path depth. Extending an extension still files the grandchild's journal at <base>/itx/<id> — the base is the journal path with its own /itx[/<id>] tail removed. Parentage lives in the birth certificate, not in path nesting.

Section 8

Code is a capability

A { type: "source" } worker ref carries code as data: the platform materializes it into an isolate on demand. Inside that isolate, bare fetch() IS project egress (secrets substituted server-side, invisible to the code) and env.ITERATE is an itx scoped to the capability's home context — a capability can never reach wider than where it is provided. Two source kinds:

Repo sources build per commit, never per call

build(repo@sha, path, bundle) → modules is a pure function, so its output is a memo, never an address. Three tiers, one key: the repo (the authority) → R2 (ITX_BUILD_CACHE, hash-keyed immutable bundles, evictable because every entry is reproducible from its key) → the Worker Loader's isolate cache (ephemeral). The memo key is the content address: build/<sha256(projectId, repo, commitOid, path, bundle, compatibilityDate)>. Bundling runs in-process on an in-memory vfs (repo DO readTree → @cloudflare/worker-bundler / esbuild-wasm → R2) — no clone, no workspace, no filesystem. A "latest" commit resolves to a sha at most once per 10-second window per isolate; concurrent dials of a cold key share one build instead of stampeding; a failed memo write never fails the dial that just built good code.

Section 9

Self-description: describe() is the agents' sense organ

Agents are a first-class audience, and their only sense organ is describe(). It always works, on every handle, and returns every breadcrumb needed to explore from here: the context, the access set, and the merged capability chain — each entry with its provenance and its provider-supplied instructions (prose, for whoever finds the cap) and types (TypeScript declarations, for machines and editors). Both ride the provide, are journaled with it, and come back from describe() on every handle down the chain. The review question for anything in this system is always the same: what does describe() say?

The describeItx provide-time hook

Instructions and types belong in the journaled meta at provide time — and the target knows its own surface best at exactly that moment. So: an rpc provide with no caller-supplied types is dialed once and asked describeItx (a reserved protocol path segment, so user capability paths can never collide with it). Explicit caller values always win; the probe is best-effort and side-effectful (an OpenApiClient probe fetches its spec) with a short deadline (3 s; 15 s for loopbacks, whose first probe may ride a cold chain) — any failure or timeout just means an unenriched provide. Probe-derived strings are capped at 64 KB (they are journaled forever). Only call-implementing targets can answer; member-shaped source caps hit the replay's reserved-name gate, which is the collision protection doing its job.

Recipe: an authenticated MCP server, credential-free addresses

The full secret story in one recipe. Material is stored once via the secrets default; from then on only a placeholder travels — 'Bearer getSecret({ key: "…" })' — and substitution happens on egress, inside the stateless terminal pipe. The session, describe(), and the journal never see the material.

Mechanics behind it: McpClient is an ordinary loopback RPC target (MCP is a client implementation, not a transport — deliberately not a capability kind). Its streamable-HTTP transport is stateless: connect → call → close per invocation, all HTTP through the cap's own itx.fetch. Its describeItx answers {} — an MCP surface is dynamic, listTools() is the discovery door. And the placeholder design has a security property worth repeating from §5: a live fetch shadow receives placeholders unsubstituted — substitution exists only in the default pipe, so an interceptor never sees secret material.

Two more describe-adjacent doors

  • HTTP exposure: a cap provided with meta: { http: { expose: true } } gets its own hostname — https://{cap}--{project}.{base}/… → ItxCapabilityIngress: 404 unless exposed; the gate is admin bearer, a signed share URL, or meta.http.public; then one core dispatch with [...capabilityPath, "fetch"].

The Laws — the invariants that ARE the architecture (README, condensed):

1. The journal holds the facts; the node holds live refs; code holds composition.
2. Props carry identity, never composition or authority-by-content.
3. Auth happens at connect, nowhere else — which context your handle points at is the authority.
4. Narrowing is construction: itx.projects.get() and itx.extend() return new handles on narrower contexts, never flags on wider ones.
5. All policy-governed egress flows through one pipe; secret placeholders are substituted only in the stateless terminal pipe.
6. One calling convention: target.call({ path, args }) — the members/path-call choice lives at the edges, never in stored data.
7. Cap'n Web terminates in the stateless worker, never in a DO.

Section 10

Extending the platform: how to build the next domain object

itx is the reference implementation of a standing doctrine (docs/domain-objects-and-stream-processors.md) for designing any durable domain object on this platform — agents, contexts, MCP sessions, whatever comes next. A context is a domain object whose state is its capability table and whose creation event is context-created. The rules:

  • Creation is an event. A durable thing's existence is not configuration — it is the FIRST event in its own stream: mint the id, append the birth certificate, return a handle. No initialize() RPC, no idempotency keys as a correctness mechanism; exactly-once is a property of the fold (first creation event wins, later ones are inert). Lazy materialization: the object comes alive on first dispatch by reading its own journal.
  • Derive what names carry; reduce everything else. DO names are structured records (project id, path). Identity, journal ref, and self-address are projections of the name — never configuration. Anything a name cannot carry arrives as an event.
  • State is a fold; the checkpoint is disposable. Storage holds { offset, state } as a cache of reduce over the journal. Delete it and replay rebuilds it. The stream is the only authority.
  • Side effects live in processEvent, never in reduce. Replay rebuilds state without re-running effects (sideEffectsAfterOffset); serialized batches give in-order execution; a requested/completed event pair makes at-least-once reruns detectable.
  • Self-describing, always. Every domain object answers describe(); every provided surface carries instructions and types from the moment it exists. The acceptance test: an agent with no prior knowledge, handed a stub, acts competently from describe() alone.

Worked example: designing CronSchedule

A hypothetical new domain object — recurring scheduled script runs for a project — designed step by step under the rules. (Hypothetical: this does not exist in the codebase; it is the doctrine applied.)

Step 1 — the structured name. One Durable Object per schedule, its name the stream coordinate: <projectId>:/cron/<cron_…>. Identity (cron_…, the last path segment), journal ref ((projectId, "/cron/<id>")), and self-address ({ type: "rpc", worker: { type: "durable-object", binding: "CRON", name } }) are all parses of the name. Note the path does not claim the reserved itx segment — that belongs to context journals. If the platform keeps a D1 row mapping bare cron_… ids to coordinates, it is a directory, never the authority.

Step 2 — the journal events, birth certificate first.

Creating a schedule is one append of schedule-created — no initialize RPC, nothing touches the new DO; it materializes when the first dispatch (or its alarm) makes it consume its journal. Payload schemas stay deliberately loose, so a journal carrying events from before a deploy never wedges the fold.

Step 3 — the reduce (pure, module-level, defensive).

Step 4 — side effects in processEventBatch, never in reduce. The alarm handler appends fire-requested { enqueued: true }; processEventBatch sees it (past sideEffectsAfterOffset, still pending in batch-final state), runs the script — the side effect — and appends fire-completed. A replay rebuilds pendingFires without re-running anything; a crash between the pair leaves a pending entry that the next wake detects and re-runs (at-least-once, detectable). Setting the next alarm is also a processEvent concern, derived from folded state.

The review checklist

  • What does describe() say about it? Could an agent with no prior knowledge act competently from that alone?
  • Is creation exactly one append, with the birth certificate as event #1? Is there any initialize RPC, setup path, or "configured" residue replay cannot reproduce?
  • Can you delete the checkpoint right now and lose nothing?
  • Does anything write state outside the journal? (If it needs to be auditable, is it an event — because events are the only writes?)
  • Do retried appends and duplicate deliveries reduce to no-ops in the fold, without idempotency keys as a correctness mechanism?
  • Are all side effects in processEvent/processEventBatch, guarded by a requested/completed pair — never in reduce?
  • Does the name carry exactly identity (and the journal coordinate), with everything else reduced from events? Is any D1 row strictly a directory?
  • How does it become dialable — a loopback scoping by dial-injected props, or an allowlisted DO namespace designed for itx:<projectId>:<name> scoping?
  • Does its stream path stay clear of the reserved itx segment?
Section 11

Glossary

context
An addressable node that holds named capabilities. Contexts form a prototype chain (child → parent with shadowing); a context may be durable (a DO with a journal) but durability is not part of the concept. ContextRef: "global", a project id (prj_…; proj_… legacy), or a child id (itx_…).
capability
A name (a path, really) plus a CapabilityTarget — either a live stub someone handed the context, or a serializable address of where the implementation lives. Its kind is its provider's kind: "live" | "rpc" | "url".
handle (ItxHandle)
The cheap, ephemeral view user code touches — identical in browser, Node, REPL, project worker, scripts, and capabilities. The trust kernel (the four verbs plus extend, super, streams, projects, project, fetch, capability(), shareUrl) plus a fallthrough proxy: any unknown name becomes a PathProxy whose terminal call is one invoke on the node's core. Minted exactly three ways: connect, narrowing, platform wiring.
stub
A live RPC reference to an object in another process (capnweb or Workers RPC). "An Itx holds capabilities; everything else holds a stub of one." ItxStub is the four-verb context protocol shape.
core (Itx)
The capability core a context node embeds (itx.ts): the path-keyed entry table, the live-stub table, longest-prefix dispatch, chain delegation — four verbs: provideCapability, revokeCapability, describe, invoke. Exposed by hosts via an itx() method (workerd doesn't pipeline through property accesses).
journal
The context's event stream — the only authority. Provides/revokes append events and self-ingest; nothing else writes state.
fold / reduce / replay
The pure function from journal to state (reduceItxJournalEvent). Replay = re-running the fold from offset 0; defensive by design so old event shapes never wedge it.
checkpoint
{ offset, state } in the host DO's storage — a disposable cache of the fold. Delete it; replay rebuilds it.
dial
The one effectful function injected into the core: CapabilityAddress → PathCallable. Owns reach (allowlists), injects attribution props, wires isolates and facets.
chain
The parent links contexts resolve through: itx_… → project → defaults (code). A lookup miss delegates the whole path upward, carrying origin.
defaults
The code-rooted final link of every chain: PlatformContext, read-only, answering the context protocol from PLATFORM_PROJECT_CAPABILITIES (ai, fetch, streams, secrets, repos, workspace, worker). Labeled from: "defaults" in describe. Shipping one is a deploy, not a migration.
extend
itx.extend({ name? }): mint a child context — an id, a journal with a birth certificate, an address, a handle. Object.create(parent) for contexts.
shadow
Providing (at a name or deeper path) on a child what the chain already resolves — your entry wins longest-prefix locally; revoke it and the inherited entry resurfaces.
super
The handle on the parent context — middleware's "call next()". A reserved word, so dot it, don't destructure it.
live vs durable
Live: an inbound stub, session-bound, in-memory, journaled only as an event (describe shows connected). Durable: a serializable address, restored at invoke time. Everything writable is durable; the root of every chain is code.
address (CapabilityAddress)
{ type: "rpc", worker, entrypoint?, props? } or { type: "url", url, headers? } — the system's one serializable data structure; context nodes are addressed with it too.
sturdy ref
Cap'n Proto's term for a serializable token restorable to a live capability. The serializable address kinds are this realm's sturdy-ref format — pure names, zero authority by content; resolveItx is the restorer. The HTTP share token is the one deliberate sealed/bearer-form exception, at the edge only.
WorkerSource
Stored source for a { type: "source" } worker: inline (code in the record, cached by cacheKey) or repo (code in a project repo, built per commit). Plus entrypoint and exportType ("worker-entrypoint" stateless | "durable-object" facet).
the memo
The R2 build cache for repo sources: build/<hash(projectId, repo, sha, path, bundle, compatibilityDate)> → built modules. An optimization, never an authority — every entry is reproducible from its key.
birth certificate
context-created { id, name, parent: { id, address } } — the first event in a journal. The fold takes the first and ignores later ones; parentage and descriptor derive from it. Code-born contexts (the project context, the defaults) have none.
stream coordinate
(namespace, path) — where a stream (hence a journal, hence a context's identity) lives. The generic host's DO name is the coordinate verbatim: <namespace>:<journalPath>.
reserved segment
Three reservations, by position: the stream path segment itx (context journals only); RESERVED_PATH_SEGMENTS (protocol/prototype names like then, call, constructor, describeItx — never traversable anywhere in a path); RESERVED_CAPABILITY_NAMES (the trust-kernel builtins — banned as a first segment only; fetch and streams deliberately not on it, because shadowing them is the point).
describeItx
The reserved provide-time self-description probe: an rpc provide with no caller types is dialed once and asked; a call-implementing target may answer { instructions?, types? }, journaled into meta (64 KB cap, best-effort, 3 s/15 s deadline).
placeholder / substitution
'Bearer getSecret({ key: "X" })' in an outbound header. Material lives in project secrets; substitution happens only in the stateless terminal egress pipe; journals, describe(), isolates, and fetch shadows only ever see the placeholder.
allowlist
DialableTargets — which bindings/loopbacks/DO-namespaces an rpc address may dial. Hardcoded defaults ∪ deployment config; config can only widen; checked at dial time (first invoke), never at provide.
facet
A Durable Object instantiated inside the DO hosting a context (ctx.facets), with its own private SQLite stored in the host. How exportType: "durable-object" source caps get state with zero provisioning; named cap:<name> (no cache key) so data survives code upgrades.
origin (ItxOrigin)
{ id, address } of the context a delegated call started at. Set by delegating nodes, never by handles — the chain's trusted identity channel for attribution, context-scoped caps, and origin dial-back.
PathCall / longest-prefix / member replay
The wire shape { path, args }; resolution picks the longest provided prefix and dispatches the remainder; targets that don't implement call get the remainder replayed onto their members, receiver-preserved, in the process where they live.
narrowing / access set
itx.projects.get(id) — the access check, returning a NEW handle bound to the project. Access is "all" (admin) or a list of project ids; missing and forbidden are byte-identical NOT_FOUND. Narrowing is construction, never a flag.
ItxProps
The one serializable parameterization crossing boundaries: { context, contextAddress?, projectId?, access?, capabilityPath? }. Identity and attribution only — never composition, never authority-by-content.
project worker
The worker built from the project's own config repo (slug project) — the worker default, an ordinary repo source (PROJECT_WORKER_SOURCE). The platform's only guarantee is that the repo exists.

Where this page simplifies (the toy vs the real core):

  • The toy is synchronous and in-process; the real verbs are async and cross RPC boundaries (capnweb and Workers RPC), with dup-retention of live stubs, member-walking retention for plain-object providers, and onRpcBroken teardown.
  • The toy's append/catch-up is a simple loop; the real seam is single-flight with an append counter guaranteeing read-your-writes under concurrent writers.
  • The toy omits: script-execution events and processor-mode runs, structural address validation, the 64 KB describeItx budget and timeouts, error masking (NOT_FOUND), the access model, and shareUrl/HTTP ingress.
  • The toy's reserved-name sets are abbreviated; the real lists live in path-proxy.ts and itx.ts.
  • The demo "defaults" simulate their dispatch results in one line each; the real defaults dial real entrypoints (EgressPipe, StreamsCapability, …) in-process.
  • The toy handle is only the fallthrough proxy; the real handle adds the built-in members and is minted via connect/narrowing/wiring.
