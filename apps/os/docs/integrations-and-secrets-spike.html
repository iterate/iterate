Integrations & Secrets

01The picture

Everything that happens in the outside world is captured first, interpreted later. Provider events land verbatim on a global journal; routing, project state, and agent fan-out are all folds over journals, downstream of the durable ack.

Webhook ingress — a partial fetch function per integration

Each provider file handles its own requests imperatively: verify signature (GitHub HMAC, Discord ed25519), answer handshakes (PING→PONG), then call the ONE side-effect primitive: capture(...).

The pattern throughout: creation is an event, state is a fold, the journal is the only write authority. Durable objects are folds plus a clock (alarms) plus, in the Secret DO's case, the one place plaintext exists.

03The journey of a GitHub webhook

  1. POST /api/integrations/github/webhook hits the OS worker. The worker tries each integration's partial fetch function; GitHub's matches, verifies x-hub-signature-256, and rejects or proceeds.
  2. capture() appends the raw body verbatim to {global}:/integrations/github/webhooks with the delivery GUID as idempotency key. That single durable append gates the 200 — provider retries never queue behind cold durable objects (the Slack latency lesson).
  3. The ingress-router DO's processor wakes (behind the ack), looks up installation:1234 in its event-sourced routing table, and cross-posts the event to {projectId}:/integrations/github — pre-warming the project's integration DO in parallel.
  4. The project's IntegrationDurableObject folds it (counters, last-event-at) and hands it to the fan-out seam — where "route this issue comment to an agent thread stream" plugs in, exactly like the Slack thread router today.

Connecting is the mirror choreography, identical for every provider and ownership mode — three appends (connect.ts): each credential → secret/set; the connection → integration/connected; each routing-key claim → route-registered on the global stream.

04Secrets: facts, deriveds, and the unifying theory

A Secret's material is either a fact (a password, a PAT, a refresh token, a plain config variable — sensitivity: "plain" covers the Doppler use) or derived: recomputed from other secrets via an HTTP exchange declared in its journal. OAuth refresh is not special — it is one derivation. So is "exchange my Waitrose username/password for a 5-minute session token". 

// the whole OAuth refresh story, as data — refresh token & client secret are sibling Secrets:
derivation: {
  kind: "http-exchange",
  request: {
    url: "https://oauth2.googleapis.com/token", method: "POST",
    body: 'grant_type=refresh_token&refresh_token=getSecret({ key: "google/refresh-token" })' +
          '&client_secret=getSecret({ key: "google/oauth-client-secret" })&client_id=…',
  },
  extract: { materialPointer: "/access_token", expiresInPointer: "/expires_in" },
}

Freshness, twice over. Every dereference — fetch-with-substitution, egress placeholder resolution, platform reveal — goes through ensureFreshMaterial: a stale token re-derives inline, on the very request that found it stale. An alarm at expiry-minus-leeway keeps hot secrets fresh proactively. Every run is journaled as secret/rotated.

Derivations chain. A derivation's getSecret({key}) references resolve through the source secrets' own DOs, which freshen themselves first — a token derived from a token derived from a password, lazily, hop by hop, with a secret/used audit event at every hop. The placeholder language is the same one egress speaks: derivation is egress substitution, one hop down.

Material never leaves the DO in the normal path. Events carry AES-256-GCM envelopes only (journals replicate freely). Callers get fetchWithSecret (substitution inside the DO) or the placeholder convention. The audited revealForPlatformUse trapdoor exists for platform-trusted consumers that need bytes in hand — SDK constructors in the integration DO, Discord's identify frame — and the websocket identify case is precisely what forced it: fetch substitution can't reach inside a WS frame.

05The Waitrose case — a userspace integration, end to end

A customer defines a whole integration in their project worker, and itx.integrations.waitrose simply exists. Waitrose has no refresh tokens: you re-login with username/password and sessions last ~5 minutes — the perfect derived secret.

  1. Connect once (per account). itx.worker.connectWaitrose({username, password, account}) journals three Secrets via itx.secrets.set: the username (plain), the password (fact), and waitrose/{account}/access-token — born material-less, carrying only its derivation (the generateSession GraphQL mutation referencing the other two, ttlSeconds: 300). Connecting mum's login too is the same call with account: "mum".
  2. Call the SDK. itx.integrations.waitrose.searchProducts("milk") forwards to the project worker's integrations({slug, account, path, args}) export as one method call (workerd RPC doesn't traverse instance fields; the path walks locally where the SDK is concrete) — itx.integrations["waitrose/mum"] reaches the second account's SDK.
  3. Egress substitutes. The SDK's bare fetch() carries authorization: Bearer getSecret({key: "waitrose/access-token"}). The terminal egress pipe asks the token's Secret DO for material; finding none, the secret derives inline — logging in with the journaled credentials — and the search leaves with a fresh Bearer token. The SDK never saw it.
  4. Six minutes later, the next search finds the token stale and re-derives transparently. The journal shows: one secret/set, two secret/rotated, audited uses throughout. (waitrose-userspace.test.ts exercises exactly this.)

Promotion path: userspace → registry is "move the provider file". The two-tier secret groundwork (tier: "project" | "iterate") maps onto two egress substitution hops: project-tier material is invisible to the customer's agents; iterate-tier material is invisible to customers and their agents.

07The itx surface

// platform integrations — the well-known SDKs, ready-authenticated:
await itx.integrations.github.octokit.rest.issues.create({ owner, repo, title });
await itx.integrations.discord.api.channels.createMessage(channelId, { content: "hi" });

// ACCOUNTS — the address IS the journal path under /integrations
// (bare slug = account "default"; a project can hold many accounts;
//  the same coordinate via itx.streams.get("/integrations/google/jonas") is the journal):
await itx.integrations["google/jonas"].gmail.users.messages.list({ userId: "me" });

// userspace:
await itx.integrations.waitrose.searchProducts("milk");
await itx.integrations["waitrose/mum"].searchProducts("biscuits");

// secrets (reveal-free: set, describe, fetch-with-substitution — never material):
await itx.secrets.set({ slug: "waitrose/password", material });
await itx.secrets.set({ slug: "waitrose/access-token", derivation: { … } });
await itx.secrets.describe({ slug: "waitrose/access-token" });  // status, version, audit — no bytes

// and inside ANY project code, the placeholder convention:
await fetch(url, { headers: { authorization: 'Bearer getSecret({ key: "waitrose/access-token" })' } });

08Open questions for the design session

  • Gateway economics: a client websocket pins its DO awake permanently. Accept per-provider? Containers? A shared gateway pool?
  • Audit volume: every dereference appends secret/used — unbounded journal growth for hot secrets. Sample, fold counters, or separate audit stream?
  • SDK calls serialize through the per-(project, integration) DO. Fine for legibility now; a hot integration can move the SDK surface back to a stateless loopback that consults the DO's fold.
  • Script derivations: declared but not executable — the escape hatch where project code computes {material, expiresAt}. Wire via the project worker?
  • Replay-on-claim: routing claims are forward-looking; the capture stream has everything needed to retroactively forward on a late connect.
  • Slack/Google migration — done. Both are provider files; Slack's thread router is the slack-route processor on the account stream (slack-agent downstream unchanged); Google's access token is a derived Secret. The legacy D1 tables are dropped; OAuth state is a signed stateless token. Existing prd journals reference the deleted SLACK_INTEGRATION namespace — prd gets a stage reset, not a bridge.
