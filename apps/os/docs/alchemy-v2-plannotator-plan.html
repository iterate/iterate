Architecture

The migration is a deploy-system rewrite, not an app-runtime rewrite. Worker handlers, Durable Objects, stream code, oRPC, and TanStack routes still execute as normal Cloudflare Worker modules.

Before

  • v1 top-level await alchemy(...)
  • alchemy/cloudflare resources
  • patched Alchemy for Artifacts
  • Vite auxiliary-worker manifest
→

Now

  • Alchemy.Stack("os", ...)
  • Cloudflare.Worker and Cloudflare.Vite
  • Cloudflare.Artifacts native binding
  • Cloudflare.QueueConsumer
→

Runtime Boundary

  • Effect stays in alchemy.run.ts
  • Workers keep async fetch
  • Env types use InferEnv
  • AI uses a raw native binding shim

Local Development

The user-facing command remains pnpm dev. Under the hood it now launches alchemy dev ./alchemy.run.ts --stage <doppler-config> through Doppler.

What the developer does

  • Run pnpm dev from the repo root or apps/os.
  • The script resolves the Doppler config, normally dev_jonas here.
  • It writes apps/os/.alchemy/dev-server.json with the chosen localhost URL.
  • The browser opens the local app URL, for example http://localhost:52011.

What changed underneath

  • Alchemy v2 provisions or reads cloud-backed resources during local dev.
  • Local Workers run in Alchemy's v2 local runtime rather than the old generated wrangler topology.
  • There is no .alchemy/local/aux-workers.json Vite integration anymore.
  • CI=1 is passed so Cloudflare auth uses Doppler credentials non-interactively.

What gets easier

  • One deploy primitive covers dev, previews, and production.
  • Native Artifacts and QueueConsumer remove local patch surface.
  • The app Worker can be typed directly from Alchemy v2 resources.
  • First-run adoption is expressed with resource policy instead of ad hoc v1 flags.

What gets harder

  • Local dev now depends on Cloudflare credentials and reachable cloud APIs.
  • Remote-backed D1/R2/AI behavior is less isolated than the old fully-local default.
  • Alchemy v2 beta currently logs spawn EBADF for some service Workers in this topology.
  • Workers AI needs a raw { type: "ai" } binding until Alchemy exposes a public marker.

Demonstrated locally: pnpm --dir apps/os dev selected http://localhost:52011. A probe of / followed the expected 307 redirect to /sign-in?redirect_url=%2F and returned 200 with a 9601 byte SSR HTML response containing the sign-in UI.

Preview And Production

Previews and production should keep the repo's current Doppler-driven environment contract: config selects secrets, Cloudflare account, app hostnames, and the Alchemy stage.

1

Doppler config

  • dev_*, preview_N, and prd set ALCHEMY_STAGE.
  • App config is serialized into APP_CONFIG.
2

Stack apply

  • Resources are declared in one Alchemy.Stack.
  • Durable Object namespaces, D1, queues, R2, loader, and Workers deploy together.
3

Routes

  • v2 lacks the old route helper we used.
  • OS keeps a custom EnsureWorkerRoutes action for DNS and routes.
4

Adoption

  • No backwards compatibility layer is kept.
  • First preview/prod run must be treated as explicit resource adoption.
5

Verification

  • Use the app CLI against preview slots after deploy.
  • Confirm route ownership, queue consumer attachment, and Durable Object state.
