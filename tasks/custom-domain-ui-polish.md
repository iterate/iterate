# UI polish for allowing customers to bring their own domain

Smooth onboarding flow for custom domains: Domain Connect for automatic DNS setup, plus a manual fallback with DNS record display and validation.

## Background: how DNS setup works today

When a user sets a custom domain (e.g. `kaletsky.com`) on their project, they need to:

1. **2x CNAME records** — routing traffic to Cloudflare:
   - `kaletsky.com` CNAME `cname.iterate.app` (apex)
   - `*.kaletsky.com` CNAME `cname.iterate.app` (wildcard, for port subdomains like `4096.kaletsky.com`)

2. **2x TXT records** (for non-Cloudflare DNS only) — DCV (Domain Control Validation) for CF for SaaS SSL cert issuance:
   - `_cf-custom-hostname.kaletsky.com` TXT `<uuid>` (CF custom hostname ownership proof)
   - `_acme-challenge.kaletsky.com` TXT `<token>` (Let's Encrypt / Google Trust DCV)

   These values are generated by the CF Custom Hostnames API when we register the hostname. If the domain's DNS is on Cloudflare, DCV is automatic and no TXT records are needed.

Currently the settings UI just shows static text ("Point these to cname.iterate.app"). This task makes it smooth.

## How Domain Connect works

Domain Connect is an open protocol (IETF draft) for automatic DNS provisioning between Service Providers (us) and DNS Providers (GoDaddy, Cloudflare, IONOS, Namecheap via third-party, etc.).

### The protocol (synchronous flow)

1. **Discovery**: query `_domainconnect` TXT record on the domain. It returns the DNS provider's Domain Connect settings URL (e.g. `_domainconnect.kaletsky.com` → `api.domainconnect.godaddy.com`).
2. **Settings fetch**: GET `https://<settings-host>/v2/<domain>/settings` → returns JSON with `providerName`, `urlSyncUX` (the apply URL), `urlAPI`, etc.
3. **Redirect to apply**: redirect the user to `<urlSyncUX>/v2/domainTemplates/providers/<providerId>/services/<serviceId>/apply?domain=<domain>&CNAME1=cname.iterate.app&...&sig=<signature>&key=<keyId>`. The DNS provider shows a confirmation UI.
4. **User confirms**: DNS provider applies the template records, then redirects back to our `redirect_uri`.

### Our template

We'd publish a Domain Connect template (`iterate.app.custom-domain.json`) to the [Domain-Connect/Templates](https://github.com/Domain-Connect/Templates) repo:

```json
{
  "providerId": "iterate.app",
  "providerName": "Iterate",
  "serviceId": "custom-domain",
  "serviceName": "Custom Domain",
  "version": 1,
  "logoUrl": "https://os.iterate.com/iterate-logo.svg",
  "description": "Connect your domain to Iterate",
  "syncPubKeyDomain": "domainconnect.iterate.app",
  "syncRedirectDomain": "os.iterate.com",
  "records": [
    {
      "type": "CNAME",
      "host": "@",
      "pointsTo": "cname.iterate.app",
      "ttl": 3600
    },
    {
      "type": "CNAME",
      "host": "*",
      "pointsTo": "cname.iterate.app",
      "ttl": 3600
    }
  ]
}
```

Note: Domain Connect can only set CNAME records for us. The DCV TXT records are handled by CF for SaaS automatically when the domain's DNS is set up correctly.

### DNS provider support

Domain Connect is supported by these DNS providers (confirmed live):

- **Cloudflare** (sync flow only, requires template onboarding via email to `domain-connect@cloudflare.com`)
- **GoDaddy** (sync + async, largest Domain Connect adopter)
- **IONOS** (1&1)
- **NameSilo**
- **WordPress.com**
- **Plesk**

**NOT supported** by: Namecheap (no Domain Connect), AWS Route53, Hetzner, OVH, Porkbun, Gandi, Hover.

This means Domain Connect will work for a portion of users but we need the manual fallback for everyone else.

## Implementation plan

### Phase 1: Domain Connect template + signing setup

1. **Generate keypair** for signing Domain Connect requests (RSA, published as TXT at `domainconnect.iterate.app`).
2. **Write template** `iterate.app.custom-domain.json` as above.
3. **Submit PR** to [Domain-Connect/Templates](https://github.com/Domain-Connect/Templates).
4. **Onboard with Cloudflare** — email `domain-connect@cloudflare.com` with template link, logo, and request proxied CNAME records.
5. **Create DNS TXT record** for `domainconnect.iterate.app` with the public key.

### Phase 2: Backend — Domain Connect discovery + apply URL builder

6. **Discovery endpoint** (`backend/services/domain-connect.ts`):
   - Input: domain name
   - Query `_domainconnect.<domain>` TXT record via DNS-over-HTTPS (e.g. `https://cloudflare-dns.com/dns-query?name=_domainconnect.kaletsky.com&type=TXT`)
   - If TXT record exists → fetch settings from `https://<value>/v2/<domain>/settings`
   - Return `{ supported: true, providerName, applyUrl }` or `{ supported: false }`

7. **Apply URL builder** — construct the signed redirect URL:
   - Template params: just the CNAME target (`cname.iterate.app`), no dynamic vars needed since our records are static
   - Sign the URL with our private key
   - Include `redirect_uri` back to our settings page with `?domain_connect=success` param

8. **oRPC endpoint** `project.checkDomainConnectSupport` — calls discovery for the project's custom domain, returns whether Domain Connect is available and for which provider.

### Phase 3: Backend — DNS validation endpoint

9. **oRPC endpoint** `project.checkCustomDomainDns`:
   - Input: custom domain
   - Queries via DNS-over-HTTPS:
     - `<domain>` CNAME → should resolve to `cname.iterate.app`
     - `*.<domain>` CNAME → should resolve to `cname.iterate.app` (may not be queryable directly; check A/AAAA resolution against CF IPs as fallback)
     - `_cf-custom-hostname.<domain>` TXT → if CF for SaaS requires it
     - `_acme-challenge.<domain>` TXT → if CF for SaaS requires it
   - Returns status per record: `{ cname: "ok" | "missing" | "wrong", wildcard: "ok" | "missing", dcvTxt: "ok" | "missing" | "not_required", acmeTxt: "ok" | "missing" | "not_required" }`

10. **Get DCV values from CF API** — after registering the custom hostname, the CF Custom Hostnames API returns the DCV TXT record values. Store these (or fetch on demand) so the UI can display them.

### Phase 4: Settings UI — custom domain setup flow

The custom domain section in project settings (`apps/os/app/routes/proj/settings.tsx`) should be a multi-step flow:

#### Step 1: Enter domain

- Text input for domain name (existing)
- "Save" button → calls `project.update` → triggers CF for SaaS registration (outbox event)
- After save, show Step 2

#### Step 2: Configure DNS

Two paths, shown as tabs or auto-detected:

**Path A: Domain Connect (automatic)**

- If `checkDomainConnectSupport` returns supported:
  - Show: "Your DNS provider ({providerName}) supports automatic setup"
  - Button: "Configure DNS automatically" → redirects to Domain Connect apply URL
  - On redirect back with `?domain_connect=success` → show success state, trigger DNS check

**Path B: Manual DNS setup (fallback)**

- Always available, default if Domain Connect not supported
- Show a table/card of required DNS records:

  | Type  | Name                               | Value               | Status |
  | ----- | ---------------------------------- | ------------------- | ------ |
  | CNAME | `kaletsky.com`                     | `cname.iterate.app` | --     |
  | CNAME | `*.kaletsky.com`                   | `cname.iterate.app` | --     |
  | TXT   | `_cf-custom-hostname.kaletsky.com` | `ca3-abcdef...`     | --     |
  | TXT   | `_acme-challenge.kaletsky.com`     | `xyz123...`         | --     |
  - TXT records: only shown if needed (non-Cloudflare DNS). Values fetched from CF API after custom hostname registration.
  - Each row has a copy button for the value
  - Note about Namecheap/registrar host field format (just `@`, `*`, `_cf-custom-hostname`, `_acme-challenge` — not the full FQDN)

#### Step 3: Verify DNS

- "Check DNS" button → calls `project.checkCustomDomainDns`
- Shows green checkmark / red X per record
- Auto-polls every 30s while the page is open (DNS propagation can take minutes)
- Once all records pass → show "Domain connected" success state
- Show CF for SaaS SSL status too (pending validation → active)

#### Step 4: Active state

- When domain is fully active (DNS + SSL):
  - Show green badge "Connected"
  - Show the custom domain URL(s): `https://kaletsky.com`, `https://opencode.kaletsky.com`
  - "Remove domain" button → clears custom domain + deletes CF custom hostname

### Phase 5: Polish

11. **Error states**: handle cases where CF for SaaS registration fails, domain already registered by another zone, etc.
12. **Loading states**: registering custom hostname is async (outbox), show "Setting up..." spinner while waiting.
13. **Toast notifications**: "DNS records verified", "SSL certificate issued", "Domain connected".
14. **Help text**: link to docs explaining DNS setup for common registrars (Namecheap, GoDaddy, Cloudflare).

## Technical notes

### DNS-over-HTTPS for validation

We can't do DNS queries from a CF Worker directly. Use fetch to Cloudflare's DoH endpoint:

```
https://cloudflare-dns.com/dns-query?name=kaletsky.com&type=CNAME
Accept: application/dns-json
```

### Domain Connect signing

Synchronous Domain Connect requests must be digitally signed (RSA-SHA256). Libraries:

- [domain-connect-dnsutils](https://github.com/domain-connect/python-domain-connect-dnsutils) (Python reference)
- We'd implement signing in TypeScript using Web Crypto API (available in Workers)

### CF for SaaS DCV record retrieval

After calling `POST /zones/{zone_id}/custom_hostnames`, the response includes `ssl.validation_records` with the TXT record names and values. We should store these on the project row or fetch them on demand via `GET /zones/{zone_id}/custom_hostnames/{id}`.

### Apex CNAME limitations

Some DNS providers don't support CNAME on the apex (root domain). Domain Connect's CNAME record with `host: "@"` handles this — the DNS provider translates it to ALIAS/ANAME/CNAME flattening as appropriate. For the manual flow, we should note this limitation and suggest the user check if their provider supports it (most modern providers do: Cloudflare, Namecheap, GoDaddy, Route53 via ALIAS).

## Open questions

1. **Template onboarding timeline**: Cloudflare reviews templates manually. How long does this take? Should we launch manual flow first and add Domain Connect later?
2. **Domain Connect for non-Cloudflare DNS providers**: some providers (GoDaddy) are the DNS host themselves, so Domain Connect works even if the domain isn't on Cloudflare. But for the DCV TXT records, users on non-Cloudflare DNS still need to add them manually (Domain Connect template can't include dynamic DCV values since they're per-hostname). Should we have a separate Domain Connect template for the CNAME records only, and show DCV records in the manual section?
3. **Polling frequency for DNS verification**: how aggressively should we poll? DNS propagation can take 0-48h. Consider: poll every 30s for 5 minutes, then every 5 minutes, then show "check manually" button.
4. **Multiple custom domains**: current schema is one per project. Should the UI hint that more might come later?
