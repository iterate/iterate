# Custom domain email delegation via Resend

Allow customers with custom project domains to fully delegate all email (inbound + outbound) to Iterate via Resend, so they don't need Gmail/Outlook or any other email provider.

## Background

When a customer sets up a custom domain (e.g. `customer.com`), we need to configure DNS records for email sending and receiving. Resend handles both via Amazon SES infrastructure. We already plan to use Cloudflare Domain Connect for CNAME records on custom domains — the same flow can set email DNS records (MX, TXT, SPFM) too.

## DNS records required

### Sending (outbound)

These records authorize Resend/SES to send email on behalf of the customer's domain:

| Type        | Name                             | Value                                                 | Notes                                                                                            |
| ----------- | -------------------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| MX          | `send.customer.com`              | `feedback-smtp.{region}.amazonses.com` (priority 10)  | Return-path for bounces/complaints. Subdomain configurable via Resend's `customReturnPath` param |
| TXT (SPF)   | `send.customer.com`              | `"v=spf1 include:amazonses.com ~all"`                 | Authorizes SES to send                                                                           |
| TXT (DKIM)  | `resend._domainkey.customer.com` | Per-domain public key (generated by Resend)           | Proves email authenticity                                                                        |
| TXT (DMARC) | `_dmarc.customer.com`            | `"v=DMARC1; p=reject; rua=mailto:dmarc@customer.com"` | Optional but recommended — tells receivers to reject forged email                                |

### Receiving (inbound)

This routes ALL inbound email for the domain to Resend:

| Type | Name                  | Value                                               | Notes                                                                                                            |
| ---- | --------------------- | --------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| MX   | `customer.com` (root) | `inbound-smtp.{region}.amazonaws.com` (priority 10) | Must be the lowest priority value (= highest priority) MX record. For full delegation this is the ONLY MX record |

Resend then fires `email.received` webhook events with email metadata. Email body/attachments must be fetched separately via Resend's Receiving API.

## Resend API calls

```
// 1. Create domain — returns all DNS record names + values
POST https://api.resend.com/domains
{ "name": "customer.com", "region": "us-east-1", "customReturnPath": "send" }

// Response includes records array with exact values for MX, SPF, DKIM

// 2. Trigger verification (async — Resend checks DNS)
POST https://api.resend.com/domains/{id}/verify

// 3. Enable receiving (sets up inbound MX requirement)
// Toggle via dashboard or API — then add the inbound MX record

// 4. Create scoped API key for sending from this domain only
POST https://api.resend.com/api-keys
{ "name": "customer.com-sending", "permission": "sending_access", "domain_id": "..." }
```

## Cloudflare Domain Connect for automated DNS setup

Domain Connect is an open protocol that lets service providers set DNS records on a customer's domain with user consent — like OAuth for DNS.

**Roles:**

- **DNS Provider** (Cloudflare, GoDaddy, etc.) — hosts customer's DNS, implements the consent UI
- **Service Provider** (Iterate) — publishes a template declaring which records are needed
- **Customer** — owns the domain, authorizes the record changes

**How it works:**

1. We query `_domainconnect` TXT record on customer's domain to discover their DNS provider
2. If supported (Cloudflare, GoDaddy, etc.), redirect customer to DNS provider's consent page with our template ID + variable values
3. Customer clicks "Authorize" — DNS provider creates all records from our template
4. Customer redirected back. Done.

**Template:** Domain Connect templates support all record types we need: `MX`, `TXT`, `CNAME`, `SPFM` (smart SPF merge). Many email providers already use this (Microsoft 365, GoDaddy Professional Mail, Customer.io, etc.).

**Onboarding:** We submit our template as a PR to the public [Domain Connect Templates repo](https://github.com/Domain-Connect/Templates). Once merged, DNS providers that auto-sync (GoDaddy, 1&1, etc.) pick it up automatically. Cloudflare has a separate manual approval process — email `domain-connect@cloudflare.com` with template link, logo, and proxy status preferences.

**Template design:** An `iterate.com.email.json` template with variable placeholders for per-domain values (DKIM key, MX targets). These get populated at apply-time from Resend's `POST /domains` response.

## Implementation plan

### Phase 1: Resend domain + API key provisioning (outbox)

1. When customer enables email on their custom domain, enqueue an outbox event `email-domain:setup-requested`
2. Consumer calls Resend `POST /domains` with the customer's domain, stores the returned DNS records + domain ID in a new `email_domain` table
3. After DNS is set (manually or via Domain Connect), enqueue `email-domain:verify-requested`
4. Consumer calls Resend `POST /domains/{id}/verify`, polls for verification status
5. On verification success, consumer creates a scoped Resend API key (`POST /api-keys` with `domain_id`) and stores it as a project secret (env var `RESEND_API_KEY` or similar)
6. Emit `email-domain:activated`

### Phase 2: Domain Connect template

7. Create `iterate.com.email.json` template with variables:
   - `%returnPathSubdomain%` — default `send`
   - `%mxSendTarget%` — from Resend response
   - `%spfInclude%` — e.g. `include:amazonses.com`
   - `%dkimSelector%` — e.g. `resend._domainkey`
   - `%dkimValue%` — per-domain public key from Resend
   - `%mxInboundTarget%` — for receiving, e.g. `inbound-smtp.us-east-1.amazonaws.com`
8. Submit PR to Domain Connect Templates repo
9. Contact Cloudflare to onboard the template
10. Implement Domain Connect discovery + redirect flow in the settings UI

### Phase 3: Inbound email handling

11. Create Resend webhook endpoint for `email.received` events
12. Route received emails to the appropriate project/agent based on `to` address
13. Store/forward emails as needed

### Phase 4: UI polish

14. Settings UI with DNS record status, auto-setup button, manual fallback
15. Connector-style configuration panel (possibly under Connectors section)

## DB schema additions

```sql
CREATE TABLE email_domain (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL REFERENCES project(id),
  domain TEXT NOT NULL,
  resend_domain_id TEXT,
  region TEXT NOT NULL DEFAULT 'us-east-1',
  status TEXT NOT NULL DEFAULT 'pending', -- pending | dns_set | verified | failed
  sending_verified BOOLEAN DEFAULT FALSE,
  receiving_enabled BOOLEAN DEFAULT FALSE,
  receiving_verified BOOLEAN DEFAULT FALSE,
  dns_records JSONB, -- cached records from Resend response
  resend_api_key_id TEXT, -- ID of the scoped API key
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

## Open questions

1. **Subdomain vs root domain for receiving?** If customer already uses root domain for personal email, we'd need a subdomain like `mail.customer.com`. For full delegation, root domain is correct. UI should warn about this.
2. **DMARC policy**: should we set `p=none` (monitor) initially and let customer tighten to `p=reject`? Safer rollout.
3. **Multiple email domains per project?** Start with one, extend later.
4. **Webhook endpoint**: should inbound emails go through the CF Worker (control plane) or directly to the daemon? Probably control plane → outbox → daemon.
5. **Rate limits**: Resend free plan allows 1 domain, 3000 emails/month. Need to account for customer plan limits.
